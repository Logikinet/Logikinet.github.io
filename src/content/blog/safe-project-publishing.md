## 背景

在整理历史工程与搭建 AquaLeap 的过程中，很容易产生一种冲动：

> 「让 Agent 扫一遍磁盘，把能 push 的都 push 上去。」

我拒绝把这种冲动自动化。公开仓库一旦带上错误归属、密钥或第三方代码，成本远高于「暂时 0 个新项目上传」。

## 问题

1. **归属不清**：目录在本机，不代表可以以个人名义开源。  
2. **许可证不明**：第三方片段、课程模板、合作代码混在其中。  
3. **密钥与路径**：旧配置文件可能含 Token、内网地址、本机用户名。  
4. **Agent 乐观偏差**：模型倾向于「完成任务」，而不是「拒绝发布」。  

## 约束

- 未确认 `ownership` 与许可证 → 不得加入公开作品集，更不得 push 为 Public。  
- Private 仓库地址**不得**写入静态站点数据文件。  
- 发布前必须过安全清单（见资源库 Workflows）。  
- 允许结果为 **0 个新公开仓库**；这是合格结果，不是失败。

## 方案

### 发布闸门（必须全部通过）

| 检查项 | 不通过则 |
| --- | --- |
| 是否本人原创或已获授权 | 不上传 / 仅私有 |
| 许可证是否明确 | 不公开 |
| 无密钥与敏感配置 | 清理后再议 |
| README 不夸大、不虚构指标 | 重写文案 |
| 与合作方/课程要求不冲突 | 不公开 |

### 站点层策略

- `visibility: "unpublished"`：本地可记，站上不出现。  
- `visibility: "private"`：可写简介，**禁止** `github` URL 字段。  
- `visibility: "public"`：才允许展示公开仓库链接。  

### Agent 层策略

- 禁止「扫描后自动 `git push --all`」类工具权限。  
- Hooks：push 前检查是否命中 `.env`、密钥正则、绝对路径。  
- 默认拒绝：来源字段为 `unknown` 的发布请求。

## 执行过程

1. 盘点历史项目时，先标「可否公开」而非「如何推送」。  
2. AquaLeap 作品集改为人工精选 + 字段级可见性。  
3. 资源库写入「安全发布检查清单」，可复制复用。  
4. [待补充：是否发生过「差点推送」的真实拦截案例]

## 结果

- 短期可能出现「公开仓库新增为 0」——可接受。  
- 站点仍可展示私有项目的**非敏感**说明，不泄露地址。  
- 降低 Agent 代发代码带来的不可逆风险。

## 失败与不足

- 人工闸门慢，盘点周期变长。  
- 部分老项目长期卡在 `verificationStatus: pending`。  
- 没有自动化密钥扫描流水线时，仍依赖自检。[待补充：是否引入 gitleaks 等]

## 可复用方法

1. **0 上传是合法终点**，写进发布原则。  
2. 把「能不能发」做成显式字段，而不是口头约定。  
3. Agent 的默认动作应是拒绝，而不是尝试完成。  
4. 公开页与私有笔记分离。
