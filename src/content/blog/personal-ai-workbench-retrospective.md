## 背景

Personal AI Workbench 最初意图（据立项时的设想）：

> 面向学习、论文、项目管理和 AI 协作的个人智能工作台，把「从想法到交付」放进同一闭环。

仓库状态：私有；站点作品集中有对应条目，**不公开仓库地址**。  
本文是复盘提纲，细节以代码与笔记核验为准。

## 问题

出现典型漂移：

1. **功能完成了**：页面/模块越来越多。  
2. **目标变模糊了**：说不清「每天打开它完成的一件事」是什么。  
3. **集成变重了**：为了「什么都能做」引入过多入口。  
4. **与 Agent 工程主线抢注意力**：时间被工作台本身的功能债占用。  

核心问题：

> 完成功能清单 ≠ 达成产品目标。

## 约束（复盘时遵守）

- 不编造 DAU、完成率、性能数字。  
- 不公开私有实现细节中的密钥与路径。  
- 对记不清的决策标 `[待补充]`，对实现细节标 `[待代码核验]`。  
- 承认失败与跳过，不改写成成功故事。  

## 方案（复盘框架）

| 维度 | 问题 |
| --- | --- |
| 原目标 | 一句话用户价值是什么？ |
| 实际交付 | 现在打开产品，主路径是什么？ |
| 漂移点 | 哪次迭代开始「为功能而功能」？ |
| 成本 | 维护与 Token/复杂度开销 |
| 取舍 | 砍什么、留什么、是否归档 |

## 执行过程（时间线占位）

1. **立项**：[待补充：时间与最初范围]  
2. **MVP**：学习/笔记/任务中的哪一块先做？[待代码核验]  
3. **扩张**：增加了哪些 AI 协作能力？[待补充]  
4. **偏离信号**：例如首页入口超过 N 个、没有每周真实使用。[待补充]  
5. **现状**：实验中 / 私有 / 非精选或仍精选？[待代码核验：与 `projects.ts` 对齐]

## 结果

### 做成了的

- [待补充：确实可用的模块列表]  
- 验证了「个人工作台」需求真实存在，但形态未必是一个大一统 App。  

### 没做成的

- 清晰的南北星指标与使用节奏。  
- 与 Agent Harness 主线的边界：谁是主产品、谁是实验床。  

### 对站点的影响

- 作品集中保留条目，状态「实验中」，描述避免夸大。  
- 相关经验回流到 Prompt/工作流（盘点、约束、发布闸门）。  

## 失败与不足

- 缺少「每周是否打开」的诚实记录，复盘依赖记忆。  
- 功能完成带来的满足感掩盖了目标漂移。  
- 未设「停止扩展」的触发条件。  
- 文档滞后，后人（包括自己）难判断意图。  

## 可复用方法

1. **每两周问一次**：主路径还是不是立项那句话？  
2. **功能上限**：MVP 后新增必须淘汰一个旧入口。  
3. **实验床 vs 产品** 标签分开，避免都做成半成品产品。  
4. 复盘先写失败，再写成绩，防止美化。  
5. 私有项目可以公开「教训」，不必公开代码。
