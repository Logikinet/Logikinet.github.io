## 背景

历史项目分散在多块存储介质中：

- 硬盘 A / B / C：[待补充：容量与大致年代]
- U 盘：[待补充：用途与是否仍为主备份]

目标不是「让 AI 读完所有文件」，而是用**可重复的低 Token 流程**，得到一份可决策的项目盘点表：哪些可开源、哪些仅私有、哪些应归档或删除。

## 问题

1. 目录深、仓库多，整库摘要会迅速烧掉上下文预算。
2. 仓库名与真实用途经常不一致，不能靠名称臆测。
3. 部分路径含本机用户名或绝对路径，不能进入公开笔记或网站。
4. 需要区分「能讲清楚用途」与「还没核验就不能写进作品集」。

## 约束

- 不向模型上传密钥、Cookie、`.env` 内容。
- 不把本机绝对路径写进公开站点；笔记中用 `{{磁盘标签}}/{{项目根}}`。
- 不根据文件夹名虚构项目用途。
- 不自动把所有 GitHub 仓库加入 AquaLeap 作品集。
- Token 优先：先元数据，再抽样，再深读。

## 方案

### 分层盘点（L0 → L3）

| 层级 | 输入 | 输出 | Token 倾向 |
| --- | --- | --- | --- |
| L0 介质清单 | 磁盘标签、挂载点代号 | 介质表 | 极低 |
| L1 顶层目录 | 仅一级/二级目录名 | 候选项目列表 | 低 |
| L2 仓库指纹 | `README` 前 N 行、package 名、是否 git | 用途假设 + 置信度 | 中 |
| L3 深读 | 仅对「待上架/待删除」子集 | 可写入作品集的描述 | 按需 |

### 盘点表字段（建议）

- 名称 / 路径代号  
- 状态：进行中 · 实验中 · 已完成 · 整理中 · 搁置  
- 可见性建议：public · private · unpublished  
- 归属与版权是否清晰  
- 缺什么（README、许可证、可运行说明）  
- 下一步（仅 1 条）

## 执行过程

1. **只列目录**：对每块介质做 L1，不递归读源码。  
2. **去重**：同一项目多副本时，标记「主副本」与「疑似备份」。[待补充：实际去重规则]  
3. **抽样 README**：仅对候选集读取标题与摘要段，禁止整文件倾倒。  
4. **可公开性闸门**：来源不明或版权未确认 → 不进公开作品集。  
5. **写入站点**：仅将核验通过条目写入 `src/data/projects.ts`；Private 不写仓库 URL。

## 结果

- 产出：一份可维护的盘点表（私有笔记）+ 站点上人工精选的项目条目。  
- [待补充：介质上大致项目数量区间，勿虚构精确值]  
- 站点侧已支持 `visibility` / `repositoryStatus` 等字段，私有仓只显示「私有仓库」。

## 失败与不足

- 无 README 的老项目仍需人工打开 IDE 才能确认用途。  
- U 盘间歇挂载导致盘点中断。[待补充：是否发生]  
- 模型对「同名不同内容」副本容易误合并，必须人工确认。  
- 低 Token 意味着**故意不完整**；完整性让位于可决策性。

## 可复用方法

1. 永远先 L0/L1，再决定是否 L2。  
2. 盘点输出用表格，不写长叙事。  
3. 公开站点与私有笔记分离：站点只收确认项。  
4. 路径脱敏后再给模型。
