## 背景

更强的模型不会自动带来可控交付。在个人项目与 Harness 实验中，常见失败不是「写不出来」，而是：

- 改动面失控  
- 引入无关依赖  
- 伪造链接或数据  
- 无限重试同一错误  

本文整理**约束（Constraints）、边界（Boundaries）、Hooks** 三层结构，作为可落地的收束方式。相关已发文章见《为什么 Agent 需要工程约束》；本文更侧重操作清单与 Hooks 落点。

## 问题

仅靠系统提示中的「请遵守最佳实践」无法稳定生效，因为：

1. 提示词无强制力  
2. 工具权限过大时，模型会「有能力越界」  
3. 缺少生命周期拦截时，错误结果已经落盘才被发现  

## 约束

### 工程约束（Constraints）

- 目标与验收标准必须可检查  
- 预算：步数 / Token / 时间  
- 禁止项：密钥、force push、无必要重构  

### 边界（Boundaries）

- 文件系统：可写路径白名单  
- 网络：默认拒绝或仅允许文档域名  
- 仓库：默认不 push；公开发布另闸门  

### Hooks

- 工具前：参数与权限校验  
- 工具后：结果摘要与脱敏  
- 任务结束：测试 / 类型检查门禁  

## 方案

```text
任务契约 → 边界配置 → Skills/Tools → Hooks → 验收 → 证据包
```

| 层 | 解决什么 | 典型产物 |
| --- | --- | --- |
| 约束 | 做什么、不做什么、如何算完 | 任务说明、验收列表 |
| 边界 | 能碰什么 | 路径/网络/密钥策略 |
| Hooks | 何时拦截 | 校验器、审计日志 |

## 执行过程

1. 任务启动前填写：目标、非目标、验收、预算。  
2. 收窄工具：只读探测 vs 受限写入分离。  
3. 在写文件与执行命令前挂校验 Hook。  
4. 结束后只接受「带证据」的完成声明。

## 结果

- 预期：越界改动与空想 API 减少，失败更早暴露。

## 失败与不足

- Hooks 写太重会变成第二套业务逻辑，难维护。  
- 边界过严导致任务无法推进，需要分级（探索模式 vs 发布模式）。  
- 无自动测试的仓库，结束门禁只能降级为「人工检查清单」。  

## 可复用方法

1. 先写验收，再给工具。  
2. 默认拒绝危险操作，显式允许例外。  
3. Hooks 只做横切，不做功能主体。  
4. 把「天马行空」当作权限与契约问题，而不是文采问题。
